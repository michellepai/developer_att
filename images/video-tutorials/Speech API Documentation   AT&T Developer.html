1. Introduction

The Speech API converts speech-to-text (ASR), text-to-speech (TTS) and provides a facility for custom speech-to-text grammar lists. Getting started is easy and many custom parameters are available in the header or body of an API request to unlock more powerful features.

Considerations

The RESTful Speech API provided by AT&T has the following considerations for you to keep in mind:

1.1. Audio Content

The audio payload must have a specific encoding that is specified in the Content-Type HTTP header parameter. If the Content-Type parameter does not match one of the following types, then an error is returned to the requester.

  • audio/amr: Preferred for it's smaller size.

      • Lorem ipsum dolor sit amet
  • audio/wav: More common, but larger than amr files.

The specific supported formats are:

  • WAV, 16 bit PCM, single channel, 8 kHz sampling
  • AMR (narrowband), 12.2 kbit/s, 8 kHz sampling

Audio Data limit

  • The audio file must be less than or equal to 4 minutes.
1.2. Speech Context

The supported speech contexts are as follows:

  • Business Search
  • Web Search
  • SMS
  • Voicemail to text
  • Question and Answer
  • Television
  • Generic

The speech context is specified in the HTTP headers, using the X-SpeechContext header. The speech context is used by the Speech services engine to properly format the transcribed text. It does not provide any additionally functionality such mashups or service blending. For example:

  • If you select the Generic context and count to five ("one two three four five") as a request, then the response in text would be the words one two three four five.
  • If you select the Business context and count to five ("one two three four five") as a request, then the response in text format would be the numbers 12345.
1.3. Request Payload

The request payload can be sent in one of two ways:

  • All at once: The Content-Length HTTP header must be present and have a value of the total size of the audio data.
  • Streamed: The Transfer-Encoding HTTP header must be present and have a value of chunked. The Content-Length HTTP header should not be present. The recommended chunked length is 0200 hexadecimal, or 512 decimal, bytes. A zero (0) at the end of the streaming indicates the end of the stream.
1.4. Unsuccessful Transcription

There may be situations where the speech enabler has successfully accepted the audio input, but it cannot produce any text associated with that audio. In this case, an SVC0001 error is returned within an HTTP 400 to indicate one of the following situations:

  • No Speech
  • Not Enough Speech
  • Too Much Speech
  • Too Quiet
  • Speech Too Soon
1.5. Failed Requests

The API Gateway does not retry failed requests to the speech enabler. Therefore, the developer must handle the error and retry the request in the app.

1.6. Language Support

Currently, only English transcriptions are supported with the exception of the Generic context, which is bilingual and accepts English and Spanish. The Speech service attempts to distinguish the language from the audio itself.

Provisioning

The RESTful Speech API provided by AT&T requires that you provision your app as follows.

  • You must register the Speech service for your an app account on the AT&T Developer Program web site in order to utilize this API.
The Speech to Text Method
  • Version 1 of this method is obsolete. You should modify your app to use the most recent version.
  • Accepts only single-channel audio formats.
  • Uses the X-SpeechContext header parameter to provide context for more accurate results.
  • Limited to one minute of audio per request. If the VoiceMail context is used, then the limit is four minutes of audio per request.
  • Accepts audio data file or streaming (chunked) audio data as input.
  • Most contexts support English only. The Generic context supports English or Spanish.
  • If the audio data is not understood, then in most cases the HTTP status code of 200 OK is returned and the Status parameter is set to Speech Not Recognized.
  • Failed requests are not retried by the API Gateway. Therefore, you must specify that your app handles this.
The Text to Speech Method
  • Requires the new, unique scope value of TTS.
  • Accepts plain text or XML-based Speech Synthesis Markup Language (SSML) as input.
  • Returns binary audio data.
  • Is limited to 500 bytes of text for processing.
  • Supports English or Spanish with male or female synthesized voices.
The Speech to Text Custom Method
  • Requires the new, unique scope value of TTS.
  • Differs from the Speech to Text method in that you may supply your own grammar of hints for more accurated results.
  • Uses the X-SpeechContext parameter to differentiate between GrammarList and GenericHints style request
  • Supports an extended list of 19 languajes.
2. OAuth

Our APIs use OAuth so you will need to include your OAuth token in each request. To obtain a token find your `key` and application `key` available from your My Apps page or register to get started for free.

You may authorize access to text-to-speech, speech-to-text, or custom speech-to-text by using the scopes below:

  • `TTS` for text-to-speech
  • `SPEECH` for speech-to-text
  • `STTC` for speech-to-text custom

To obtain your OAuth token run the following command from the CLI and replace `{{KEY}}`, `{{SECRET}}`, and `{{SCOPE}}` apps key, secret, and the api scope you wish to access:

Get Access Token via OAuth for Speech:

copy 
        curl "https://api.att.com/oauth/token" \
        --insecure \
        --header "Accept: application/x-www-form-urlencoded" \
        --header "Content-Type: application/x-www-form-urlencoded" \
        --data "grant_type=client_credentials&client_id={{KEY}}&client_secret={{SECRET}}&scope={{SCOPE}}"

A successful response will contain an access token, expiration period (0 for tokens which don't expire) and a refresh token: 

      {
      "access_token":"9f101ed598ca7fd975fadc2a12b85a2c",
      "expires_in":"0",
      "refresh_token":"e2bc4782d32a1357f96e3604ad79724064008006"
      }

Extract the value from `access_token` to use in the `Authorization` HTTP header to authenticate API requests. When sending an API request a successfully constructed `Authorization` HTTP header will look like this: 

Authorization: Bearer 551f90a1833e577b13dba1c28a6289c5

cURL Request

Using the `curl` CLI tool you can easily post audio data with the `-d` option. If you don't have an audio file to test with try creating one with the text-to-speech API.

copy 
curl -XPOST https://api.att.com/speech/v3/textToSpeech \
--header "Authorization: Bearer 551f90a1833e577b13dba1c28a6289c5" \
--header "Accept: application/json" \
--header "Content-Type: audio/x-wav" \
-d @hello-world.wav
Parameters

The main payload for any speech-to-text request is the binary audio data which is the contents of the request body. You can see examples of streaming and non-streaming requests in the Examples section below.

All other parameters are HTTP headers and are described below.

HTTP Header
Accept
Specifies the format of the body for the response. Options:
  • application/json
  • application/xml
  • application/emma+xml

    • Default: application/json.
Authorization
Specifies the authorization. The acceptable format for this parameter is the word "Bearer" followed by a space ( ) and an OAuth access token. If you receive an HTTP status code of 400 or 401 then your authorization token is missing or invalid, respectively
Content-Type
Specifies the content type. Options:
  • audio/amr
  • audio/amr-wb
  • audio/wav
  • audio/x-wav
  • audio/x-speex
  • audio/x-speex-with-header-byte;rate=16000
  • audio/x-speex-with-header-byte;rate=8000
  • audio/raw;coding=linear;rate=16000;byteorder=LE
  • audio/raw;coding=linear;rate=16000;byteorder=BE
  • audio/raw;coding=linear;rate=8000;byteorder=LE
  • audio/raw;coding=linear;rate=8000;byteorder=BE
  • audio/raw;coding=ulaw;rate=16000
  • audio/raw;coding=ulaw;rate=8000
Content-Length
(Required only for non-streaming requests) The length in bytes of the audio data
Transfer-Enconding
(Required for streaming only) You must set this parameter to `chunked` for streaming requests
X-SpeechContext
Specifies the speech context being applied to the transcribed text. The acceptable values for this parameter are:
  • BusinessSearch
  • Gaming
  • Generic
  • QuestionAndAnswer
  • SMS
  • SocialMedia
  • TV
  • VoiceMail
  • WebSearch
Default: Generic.
X-Arg
Specifies multiple "extra argument" parameters within a single HTTP header. Example: "X-Arg: Volume=100,Tempo=0". See X-Arg Parameters for options.
X-Arg Parameters
HTTP Header
ShowWordTokens
Increases detail of output. For formatted ResultText shows additional detail for named entitites and the words they were derived from. For unformatted ResultText shows additional detail for all words. Default: false
Content-Language
Language of the audio, `en-US` for English in the United States of America or `es-US` for Spanish in the United States of America. Default: `en-US`. This parameter is only used when the X-SpeechContext parameter value is set to Generic
Error Codes

cURL

Using the `curl` CLI tool you can easily convert text to speech and save the binary data to a file with the `-o` option.
copy 
curl -XPOST https://api.att.com/speech/v3/textToSpeech \
--header "Authorization: Bearer 551f90a1833e577b13dba1c28a6289c5" \
--header "Content-Type: text/plain" \
--header "Accept: audio/x-wav" \
-d 'Hello world!' \
-o hello-world.wav
On most Linux or Mac command lines you can then type `open hello-world.wav` and your default audio player will play the file.

Text To Speech

Get Access Token via OAuth for TTS:

Make sure to replace {{KEY}} and {{SECRET}} with your key and secret.
copy 
              curl "https://api.att.com/oauth/token" \
              --insecure \
              --header "Accept: application/x-www-form-urlencoded" \
              --header "Content-Type: application/x-www-form-urlencoded" \
              --data "grant_type=client_credentials&client_id={{KEY}}&client_secret={{SECRET}}&scope=TTS
Copy the "access_token" for the next step.

Convert text into speech

Replace {{TOKEN}} with your OAuth access token from above. You can replace "hello world!" with anything you would like to have converted and the audio file will be saved to the filename after the `-o`.
copy 
              curl -XPOST https://api.att.com/speech/v3/textToSpeech \
              --header "Authorization: Bearer {{TOKEN}}" \
              --header "Content-Type: text/plain" \
              --header "Accept: audio/x-wav" \
              -d 'Hello world!' \
              -o hello-world.wav
Speech To Text

Send Audio file for Speech To Text conversion:

copy 
                curl "https://api.att.com/rest/2/SpeechToText" \
                --insecure \
                --request POST \
                --header "Authorization: Bearer your_oauth_access_token" \
                --header "Content-type: audio/wav" \
                --header "Accept: application/json" \
                --header "X-SpeechContext: Generic" \
                --data-binary "audio_file" \
                --proxy "https://proxy.if.you.use.one.com:8080"

8. FAQs

We've compiled answers to the most frequently asked questions about AT&T's Speech API below.
Have questions that are not API specific? Visit our full FAQs page.

8.1. Do I need to develop my own UI for capturing speech?

We can help jumpstart your application development process with native SDKs for multiple mobile platforms. These SDKs will offer guidance on developing a UI for a thin client to be embedded in smartphone applications for capturing your end users' speech. We also offer Sencha and Microsoft web services SDKs. You also have the option to build your own UI, and use the speech enabler on any operating system, including web applications.

8.2. What functionality does the Speech API provide?

There are three classes of functionality in the Speech API family:

  • Speech to Text 9 contexts are optimized to return the text of what your end users say. The text can be returned in multiple formats, including, JSON and XML.
  • Text to Speech Male and female 'characters' are available for both English and Spanish.
  • Speech to Text Custom allows you to customize the speech service by sending a list of words or phrases commonly spoken by your end users to improve recognition of those unique words
8.3. Do you support languages other than English?

Yes, here are the languages we currently support in our Speech API family:

  • Speech to Text Currently, we offer automatic English/Spanish detection with the Generic speech context, and English support for our other optimized speech contexts. Other language support is on our roadmap.
  • Text to Speech Male and female 'characters' are available for both English and Spanish.
  • Speech to Text Custom Grammar List supports 19 languages, and Generic with Hints supports English and Spanish.

The following formats are supported by the AT&T Speech API. More formats will be added in subsequent releases:

  • 16 bit PCM WAV, single channel, 8 kHz sampling
  • 16 bit PCM WAV, single channel, 16 kHz sampling
  • 16 bit PCM Raw, single channel, 8 kHz sampling
  • 16 bit PCM Raw, single channel, 16 kHz sampling
  • -law PCM WAV, single channel, 8 kHz sampling
  • -law PCM WAV, single channel, 16 kHz sampling
  • -law PCM Raw, single channel, 8 kHz sampling
  • -law PCM Raw, single channel, 16 kHz sampling
  • AMR (narrowband), 12.2 kbit/s, 8 kHz sampling
  • AMR (wideband), 12.65 kbit/s, 16kHz sampling
  • OGG - speex encoding, 8kHz sampling
  • OGG - speex encoding, 16kHz sampling
  • Header-Byte - speex encoding, 8kHz sampling
  • Header-Byte - speex encoding, 16kHz sampling

If your audio file is not in one of these formats, then you can try converting it to these formats by using audio conversion tools such as Sox or the Speex library for embedded applications.

Note:

  • Although wide-band and uncompressed audio provides the best accuracy for speech recognition, various audio compression audio coders can still allow good quality speech recognition results. Among compression codecs, we have found through experiments that the speex encoder provides better quality recognition than either the u-law compander or AMR compression formats.
  • Using the raw audio or streaming speex data can significantly reduce user perceived latency over traditional WAV formats, which cannot be streamed.
